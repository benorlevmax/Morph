# NNUE Forensic Audit — 100-0 loss (Elo -2400.0) vs classical eval

Scope: exhaustive code-level, mathematical, and runtime audit of every stage between "trained network file" and "search uses its score," to determine whether the catastrophic 100-0 result is an implementation bug, a training-pipeline bug, or a genuinely weak-but-correctly-built network. Rule followed throughout: every PASS below is backed by a specific, reproducible test that was actually run, not by reading code and assuming it's fine. Anything not backed by such a test is marked NOT VERIFIED, not PASS.

## Executive summary

No implementation defect was found anywhere in the engine-side NNUE code, the accumulator maintenance, the feature indexing, the quantization arithmetic, or the search/TT integration — all of these were exercised with real, adversarial, reproducible tests (121,886 stress-tested positions, byte-for-byte cross-language comparison, exhaustive undo-to-root verification) and every one came back clean.

What *was* found is a training-pipeline configuration that is very likely to produce a severely undertrained network: by default, a single `TRAIN_NETWORK` task samples only 200,000 positions out of the (currently ~12.8M-position) corpus and trains for 6 epochs with a flat, unscheduled learning rate — roughly 4,500 gradient steps total against a ~5.24M-parameter feature transformer. That is a training budget several orders of magnitude below what NNUE nets of this size normally need, and it is consistent with a network that plays close to randomly rather than one that's broken.

**Conclusion: most likely explanation is an undertrained-but-correctly-implemented network, not an implementation bug.** Confidence: moderate-high. This is not yet certain because Phases B–D (numeric checks against the *actual* trained weights) could not be completed in this sandbox — the real `.nnue` file lives only on your machine. A script to close that gap is included below; its output would move this from "moderate-high confidence" to "confirmed."

## Findings, ranked by severity

**MAJOR — training pipeline undersamples the corpus and undertrains relative to network size.**
`platform/server/auto_pipeline.py`: `--max-dataset-positions` defaults to 200,000 and `--train-epochs` defaults to 6, regardless of how large the underlying position corpus has grown (currently ~12.8M). `train.py`'s reference trainer then runs Adam (`lr=0.01`, flat, no decay/warmup) for those 6 epochs at `batch_size=256` — roughly 4,500 total gradient steps — against a feature transformer with `10,240 × 512 ≈ 5.24M` weights, initialized from `N(0, 0.01²)`. For comparison, production NNUE nets of this size are typically trained on hundreds of millions to billions of positions over many more effective steps, usually with a learning-rate schedule. A 100-0 loss against a hand-tuned classical evaluator is exactly the signature you'd expect from a net this undertrained: it hasn't seen enough data or gradient steps to move meaningfully from its near-random initialization, so its evaluations carry little real signal. This is the leading root-cause hypothesis.

**MINOR — latent `qa`/`qa-preview` mismatch (dormant on the observed default configuration).**
`train.py --qa-preview` (used only to size the training-time clipping range, default 256) is never passed by `train_network.py`, which instead only forwards `--qa`/`--qb` to `export.py` after training completes. As long as both stay at their shared default of 256, the numbers happen to agree and nothing breaks. If a future run passed a non-default `qa`, training and export would silently disagree on the clipping range used to shape gradients versus the range used to quantize, degrading (not fatally breaking) the exported network. Recommend threading `qa`/`qb` through to `train.py --qa-preview` explicitly so this can't drift.

**No critical or implementation-level bugs were found.**

## Recommendations, in priority order

1. **Increase training scale drastically**: raise `--max-dataset-positions` well above 200,000 (ideally use the full corpus, or a much larger rotating sample) and raise `--train-epochs` substantially, or restructure to more gradient steps per position seen.
2. **Add a learning-rate schedule** (warmup + decay) to `train_reference()` in `train.py` — a flat `lr=0.01` for the entire run is unusual for this parameter count.
3. **Re-run `test.py` after retraining** and compare Elo against the classical baseline; a network trained on meaningfully more data/steps should close most of the -2400 gap even before any further tuning.
4. **Upstream `/tmp/nnue_forensic_test.cpp`'s methodology as a permanent CTest** so accumulator/serialization regressions are caught automatically going forward instead of requiring a manual audit like this one.
5. **Fix the `qa`/`qa-preview` sync** in `train.py`/`train_network.py` before ever running with non-default quantization settings.
6. Once a retrained network is available, run `real_network_audit.py` against it to confirm the saturation/dead-neuron/output-distribution numbers look healthy before spending more Elo-match compute on it.
